PRD (Product Requirements Document)

1. 제품 개요

1.1 목적

사용자가 하루 단위로 계획(Plan) 과 실행(Execution) 을
시간 기반 타임라인 UI에서 나란히 비교하여
계획 대비 실행의 차이를 직관적으로 파악할 수 있도록 한다.

⸻

2. 핵심 UX 원칙
	1.	비교 우선
	•	“계획 vs 실행” 차이를 한 눈에 인지
	2.	입력 최소화
	•	실행 기록 시 기존 계획 재사용
	3.	시간 직관성
	•	시간 길이 = 블록 높이
	4.	조작 예측 가능성
	•	드래그/리사이즈 시 시간 변화가 즉시 시각화

⸻

3. 타임라인 레이아웃 정의 (상세)

3.1 컬럼 구조

| 시간 | 계획 (Plan) | 실행 (Execution) |

	•	좌측: 시간 레이블
	•	중앙: 계획
	•	우측: 실행

⸻

3.2 타임라인 그리드 기준 (확정)

시간 레이아웃
	•	1시간 단위 표시
	•	예: 09:00, 10:00, 11:00 …

Plan / Execution 그리드
	•	30분 단위 시각적 분할
	•	1시간 = 2칸

⸻

3.3 픽셀 기준 (권장값, 조정 가능)

기준	값
1시간 높이	120px
30분 단위	60px
10분 스냅	20px
5분 스냅	10px

계산 공식
blockHeight(px) = (endTime - startTime in minutes) × (120px / 60min)

⸻

4. 시간 관리 정책

4.1 시간 단위 정책
	•	내부 데이터: 분 단위 (HH:mm 포맷)
	•	UI 스냅 단위: 10분 (고정)
		•	드래그/리사이즈 시 10분 단위로 스냅
		•	예: 09:00, 09:10, 09:20, 09:30 ...
	•	최소 블록 길이: 10분

⸻

5. 드래그 / 리사이즈 상세 규칙

5.1 블록 생성
	•	트리거: Plan / Execution 영역 빈 공간 드래그
	•	동작:
		•	드래그 시작 지점 → 시작 시간
		•	드래그 종료 지점 → 종료 시간
		•	실시간 미리보기 표시 (반투명 블록)
	•	스냅 단위: 10분
	•	최소 생성 길이: 10분

⸻

5.2 블록 이동 (Drag & Drop)
	•	트리거: 블록 전체 영역 드래그
	•	동작:
		•	시작/종료 시간 동일 간격 유지
		•	드래그 중 블록 opacity 70% + shadow
	•	스냅 단위: 10분
	•	제약:
		•	Plan ↔ Execution 컬럼 간 이동 불가
		•	동일 컬럼 내 이동만 허용
	•	시각 피드백:
		•	이동 중 블록 위치 실시간 업데이트
		•	드롭 시 최종 위치로 애니메이션

⸻

5.3 블록 리사이즈
	•	트리거:
		•	상단 핸들 드래그 → 시작 시간 변경
		•	하단 핸들 드래그 → 종료 시간 변경
	•	핸들 표시:
		•	평소: 투명 (invisible)
		•	호버 시: 반투명 (opacity 30%)
		•	높이: 8px
	•	동작:
		•	리사이즈 중 블록 높이 실시간 업데이트
		•	리사이즈 중 opacity 70% + shadow
	•	스냅 단위: 10분
	•	최소 블록 길이: 10분
	•	시간 제약:
		•	시작 시간 < 종료 시간
		•	시작 시간은 종료 시간 이상으로 조정 불가
		•	종료 시간은 시작 시간 이하로 조정 불가

⸻

6. 일정 겹침 처리

6.1 겹침 기준
	•	동일 컬럼 내
	•	시간 범위가 겹치는 경우

6.2 표시 규칙
	•	2개 겹침:
	•	수직 분할
	•	3개 이상:
	•	대표 2개 표시
	•	나머지 +N 표시
	•	+N 클릭 시:
	•	리스트 팝업 또는 모달

⸻

7. 실행 여부 & 달성률

7.1 실행 판단
	•	계획과 동일한 행동을 일부라도 수행 시 “실행”
	•	시간 완전 일치 불필요

7.2 달성률 계산

달성률 = 실행 시간 / 계획 시간 × 100

7.3 시각적 표현
	•	미실행 계획:
	•	흐린 색상 + 점선 테두리
	•	실행된 계획:
	•	달성률 배지 (예: 80%, 120%)

⸻

8. 카드 UI 규칙

8.1 색상
	•	과목 구분용
	•	사용자 지정 가능
	•	상태 표현용 아님

8.2 텍스트
	•	말줄임 처리 (…)
	•	줄바꿈 없음
	•	전체 텍스트는 툴팁/상세뷰 제공

8.3 헤더 UI 디자인

날짜 네비게이션 영역
	•	배경: 흰색 또는 연한 회색
	•	테두리: 하단 구분선
	•	높이: 60~80px (모바일), 80~100px (데스크톱)
	•	버튼:
		•	◀◀: 오늘로 이동 (Today)
		•	◀: 이전 날짜
		•	날짜: 클릭 가능한 버튼 형태
		•	▶: 다음 날짜
		•	▶▶: 마지막 날짜 (또는 비활성화)
	•	날짜 포맷: YYYY.MM.DD 또는 YYYY년 M월 D일

통계 정보 영역
	•	배경: 약간 더 연한 색상 (구분감)
	•	높이: 40~60px
	•	대표 일정 Chip:
		•	배경: 연한 파스텔 색상
		•	텍스트: 12~14px, 굵게
		•	패딩: 6px 12px
		•	둥근 모서리: 16px
		•	최대 3개 표시, 초과 시 +N
	•	통계 정보:
		•	폰트: 12~13px
		•	색상 구분:
			- 계획: 파란색 계열
			- 실행: 초록색 계열
			- 달성률: 그라데이션 또는 조건부 색상
				* 80% 이상: 초록색
				* 50~80%: 노란색
				* 50% 미만: 빨간색

⸻

9. 헤더 영역 & 날짜 네비게이션

9.1 헤더 구조

헤더는 2개 섹션으로 구분된다.

Section 1: 날짜 네비게이션 영역
	•	레이아웃: 중앙 정렬
	•	구성 요소 (좌→우):
		1.	이전 날짜 버튼 (◀)
		2.	오늘로 이동 버튼 (◀◀)
		3.	날짜 표시 + 캘린더 버튼 (2025.09.10 📅)
		4.	다음 날짜 버튼 (▶)
		5.	맨 끝 날짜 버튼 (▶▶)
	•	날짜 클릭 시:
		•	캘린더 위젯 표시
		•	날짜 선택 시 해당 날짜로 이동

Section 2: 통계 정보 영역
	•	레이아웃: 좌우 분할
	•	구성 요소:
		•	좌측: 대표 일정 (Chip 형태, 복수 가능)
			•	예: [중간고사] [1차 기말]
			•	클릭 시: 해당 일정 상세 보기
		•	우측: 계획/실행 요약
			•	계획: 7시간 30분
			•	실행: 5시간
			•	달성률: 67%

⸻

9.2 날짜 이동 방식

9.2.1 화살표 버튼
	•	◀ (이전): 1일 전
	•	◀◀ (맨 앞): 오늘
	•	▶ (다음): 1일 후
	•	▶▶ (맨 뒤): 데이터가 있는 마지막 날짜

9.2.2 캘린더 선택
	•	트리거: 헤더의 날짜 표시 영역 클릭
	•	캘린더 위젯 (Popup/Dropdown)
		•	위치: 날짜 버튼 하단 또는 화면 중앙
		•	레이아웃:
			- 상단: 월 선택 (◀ 2025년 9월 ▶)
			- 중앙: 날짜 그리드 (일~토)
			- 하단: 빠른 이동 버튼 (오늘, 취소)
		•	날짜 표시:
			- 오늘: 파란색 원형 하이라이트
			- 선택된 날짜: 진한 파란색 배경
			- 데이터 있는 날짜: 하단에 작은 점 표시
				* 계획만: 파란 점
				* 실행만: 초록 점
				* 둘 다: 2색 점
		•	인터랙션:
			- 날짜 클릭 → 캘린더 닫힘 + 타임라인 로드
			- 월 이동 화살표 클릭 → 이전/다음 월 표시
			- 외부 영역 클릭 → 캘린더 닫힘

9.2.3 스와이프 (모바일)
	•	좌 스와이프: 다음 날
	•	우 스와이프: 이전 날

⸻

9.3 대표 일정 (Date Event)

9.3.1 표시 방식
	•	형태: Chip (태그)
	•	위치: 통계 영역 좌측
	•	개수: 제한 없음 (가로 스크롤 또는 줄바꿈)
	•	스타일:
		•	배경: 연한 색상
		•	텍스트: 짧은 제목 (예: "중간고사")
		•	아이콘: 선택적 (🎯, 📚 등)

9.3.2 등록/수정
	•	날짜별 대표 일정 등록 가능
	•	일정 추가/수정은 설정 또는 상세 화면에서 진행
	•	타임라인에서는 표시만 (클릭 시 상세 이동)

⸻

10. 사용자 시나리오 (Main Scenario)

시나리오 목적

본 시나리오는 하루 단위 Plan / Execution 타임라인 화면에서 사용자가 계획을 확인하고, 실행을 기록하며, 계획 대비 실행 결과를 비교·조정하는 전체 흐름을 설명한다.

이 시나리오는 다음을 동시에 검증한다.
	•	타임라인 그리드 규칙
	•	드래그 / 리사이즈 인터랙션
	•	계획 ↔ 실행 관계 모델
	•	달성률 및 미실행 표시 UX
	•	날짜 이동 및 대표 일정 노출

⸻

메인 시나리오: “계획한 하루를 실행하고 돌아보는 흐름”

Step 1. 사용자는 오늘 날짜의 타임라인 화면에 진입한다
	•	앱 진입 시 기본으로 오늘 날짜의 Day Timeline 화면이 열린다.
	•	상단 헤더는 2개 섹션으로 구성된다.
		
		Section 1: 날짜 네비게이션
		•	중앙: 날짜 표시 (예: 2025.09.10) + 캘린더 아이콘
		•	좌/우: 화살표 버튼 (◀◀ ◀ | ▶ ▶▶)
		
		Section 2: 통계 정보
		•	좌측: 대표 일정 Chip (예: [중간고사] [1차 기말])
		•	우측: 계획/실행 요약
			- 계획: 7시간 30분
			- 실행: 5시간
			- 달성률: 67%
	
	•	화면 중앙에는 타임라인이 표시된다.
		•	좌측: 시간 컬럼 (1시간 단위 표시)
		•	중앙: 계획(Plan) 컬럼
		•	우측: 실행(Execution) 컬럼
		•	계획/실행 영역: 30분 단위 그리드 (시각적 구분)

⸻

Step 2. 사용자는 오늘의 계획을 시간 흐름에 따라 확인한다
	•	사용자는 타임라인을 스크롤하며
	•	언제 무엇을 계획했는지
	•	어느 시간대에 계획이 몰려 있는지
를 직관적으로 파악한다.
	•	계획 블록은:
	•	과목 색상으로 구분되어 있으며
	•	블록 높이는 계획 시간 길이를 그대로 반영한다.
	•	텍스트가 길 경우 말줄임 처리되어 있다.

⸻

Step 3. 사용자는 실행 기록을 추가한다 (계획 기반)
	•	사용자는 실행 컬럼의 특정 시간대(예: 17:00 ~ 18:00)를 길게 드래그한다.
	•	“실행 추가” UI가 나타난다.
	•	실행 생성 시:
	•	오늘 날짜의 계획 목록이 선택 옵션으로 제공된다.
	•	사용자는 기존 계획 중 하나를 선택한다.
	•	과목, 제목, 색상, 기본 시간 정보가 자동으로 채워진다.
	•	사용자는 실제 수행한 시간에 맞게
	•	블록을 리사이즈하여 종료 시간을 조정한다.

⸻

Step 4. 사용자는 계획과 다른 실행도 기록한다
	•	같은 시간대에
	•	계획에는 없던 다른 과목을 실행한 경우,
	•	사용자는 실행 컬럼에서 다시 드래그하여
	•	계획을 선택하지 않고
	•	실행 블록을 직접 생성한다.
	•	이 실행 블록은:
	•	계획과 연결되지 않은 독립 실행으로 저장된다.

⸻

Step 5. 실행 결과가 계획과 자동 비교된다
	•	실행이 하나라도 연결된 계획 블록은 “실행됨” 상태로 판단된다.
	•	계획 시간 대비 실행 시간에 따라:
	•	달성률이 자동 계산된다.
	•	예: 60%, 100%, 130%
	•	달성률은 계획 블록에 배지 형태로 표시된다.
	•	계획했으나 실행되지 않은 블록은:
	•	흐린 색상
	•	점선 테두리
로 시각적으로 구분된다.

⸻

Step 6. 사용자는 겹치는 일정도 자연스럽게 인지한다
	•	동일 시간대에 일정이 2개 겹칠 경우:
	•	블록은 수직 분할되어 나란히 보인다.
	•	3개 이상 겹칠 경우:
	•	대표 2개만 표시
	•	나머지는 +N으로 축약 표시된다.
	•	사용자는 +N을 눌러
	•	해당 시간대의 전체 일정을 확인할 수 있다.

⸻

Step 7. 사용자는 계획 또는 실행을 수정한다
	•	사용자는 계획 블록의 하단을 드래그하여
	•	종료 시간을 30분 늘린다.
	•	블록은:
	•	10분 스냅 단위로 조정된다.
	•	변경 즉시:
	•	계획 총 시간
	•	달성률 계산 결과
가 다시 반영된다.

⸻

Step 8. 사용자는 날짜를 이동하며 흐름을 비교한다
	•	사용자는 다음 방법으로 날짜를 이동할 수 있다.
		1.	화면을 좌우로 스와이프
		2.	헤더의 화살표 버튼 (◀ ▶) 클릭
		3.	날짜 표시 영역 클릭 → 캘린더에서 날짜 선택
	•	날짜가 변경되면 다음이 갱신된다.
		•	타임라인 (계획/실행 블록)
		•	대표 일정 Chip
		•	통계 정보 (계획/실행 시간, 달성률)
	•	캘린더에서는:
		•	계획/실행 데이터가 있는 날짜를 시각적으로 구분
		•	빠른 날짜 점프 가능

⸻

Step 9. 사용자는 하루를 “돌아본다”
	•	사용자는 오늘 하루를 보며 다음을 한 눈에 확인한다.
	•	계획한 것 중 하지 못한 것
	•	계획보다 더/덜 한 것
	•	계획에 없었지만 실제로 한 것
	•	사용자는 이를 바탕으로
	•	다음 날 계획을 더 현실적으로 조정한다.

시나리오 핵심 요약 (PRD 관점)

이 시나리오는 다음을 보장한다.
	•	비교 중심 UX: 계획과 실행의 차이가 즉시 보인다.
	•	입력 효율성: 실행 기록 시 계획을 재사용할 수 있다.
	•	시간 직관성: 블록의 크기와 위치가 곧 시간이다.
	•	유연성: 계획과 다른 행동도 자연스럽게 기록된다.
	•	확장성: 이후 주/월 단위 분석으로 확장 가능하다.
